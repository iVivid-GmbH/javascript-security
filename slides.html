01 — Cross-Site Scripting (XSS)

Attack Types

  • Stored — Attacker injects into database
  • Reflected — Attacker tricks user to click malicious link
  • DOM-based — JavaScript reads untrusted source

Attack Goals

  • Steal session cookies / tokens
  • Modify page content (phishing)
  • Redirect to malware
  • Deploy keylogger

Prevention Code

// VULNERABLE: innerHTML
elem.innerHTML = userInput;

// SAFE: textContent
elem.textContent = userInput;

// SAFE: DOMPurify
elem.innerHTML = DOMPurify.sanitize(userInput);

// SAFE: createElement
const p = document.createElement('p');
p.textContent = userInput;
container.appendChild(p);

02 — Clickjacking

How It Works

  1. Attacker creates overlay with invisible iframe
  2. Positions sensitive button on top of invisible element
  3. User clicks, triggering unintended action

Variants

  • UI redressing (fake buttons)
  • Cursoring attacks (cursor overlay)
  • Drag-to-iframe attacks

Defense Headers

// X-Frame-Options
response.setHeader('X-Frame-Options', 'DENY');

// Content-Security-Policy (modern)
response.setHeader(
  'Content-Security-Policy',
  "frame-ancestors 'none'"
);

// Helmet.js
helmet.frameguard({ action: 'deny' });
helmet.contentSecurityPolicy({
  directives: { frameSrc: ["'none'"] }
});

03 — Prototype Pollution

Vulnerable Code

// VULNERABLE: Deep merge
function merge(target, source) {
  for (let key in source) {
    if (typeof source[key] === 'object') {
      merge(target[key], source[key]);
    } else {
      target[key] = source[key];
    }
  }
}

// Attack: {"__proto__": {"isAdmin": true}}
// Now every object has isAdmin = true!

Secure Fixes

// Fix 1: Use Object.freeze
Object.freeze(Object.prototype);
Object.freeze(Object);

// Fix 2: Use Object.create(null)
const target = Object.create(null);

// Fix 3: Validate keys
const whitelist = ['name', 'email'];
for (let key in source) {
  if (whitelist.includes(key)) {
    target[key] = source[key];
  }
}

// Fix 4: Use hasOwnProperty
for (let key in source) {
  if (source.hasOwnProperty(key)) {
    target[key] = source[key];
  }
}

04–08 — More Client-Side Threats

DOM Clobbering

Overwrite globals via HTML IDs:

<img id="config">

Now config is the img, not the object.

eval()

Never execute untrusted code:

eval(userInput); // ❌ NEVER

Use JSON.parse() or Function() with constraints.

Insecure Storage

Never store secrets in localStorage:

localStorage.token // ❌

Use HttpOnly cookies instead.

ReDoS

Catastrophic backtracking:

/^(a+)+$/.test(userInput);

Use regex timeouts or simpler patterns.

Open Redirects

Validate redirect targets:

if (url.startsWith('/')) { // ✓ Safe
  window.location = url;
}

09 — SQL Injection

Vulnerable Pattern

// VULNERABLE: String concatenation
const query = "SELECT * FROM users WHERE id = " + userId;
db.query(query);

// Attack: userId = "1 OR 1=1"
// Becomes: SELECT * FROM users WHERE id = 1 OR 1=1

// Returns ALL users instead of one!

Safe: Parameterized Queries

// pg (PostgreSQL)
const { rows } = await client.query(
  'SELECT * FROM users WHERE id = $1',
  [userId]
);

// mysql2 (MySQL)
const [rows] = await connection.execute(
  'SELECT * FROM users WHERE id = ?',
  [userId]
);

// Prisma (ORM)
const user = await prisma.user.findUnique({
  where: { id: userId }
});

10–13 — More Injection Types

NoSQL Injection

MongoDB queries with untrusted input:

// VULNERABLE
db.users.find({email: req.body.email});

// Attack: {"$ne": null}
// Returns all users!

Command Injection

Executing shell commands:

// VULNERABLE
exec(`ls ${directory}`);

// Attack: directory = "; rm -rf /"

LDAP Injection

LDAP query manipulation:

// VULNERABLE
ldap.search({
  filter: `(uid=${username})`
});
// Use escaping libraries

Template/SSTI

Server-Side Template Injection:

// VULNERABLE
template.compile(userTemplate);

// Use sandboxed engines

14 — CSRF (Cross-Site Request Forgery)

Attack Example

<!-- evil.com -->
<form action="bank.com/transfer"
      method="POST">
  <input name="to" value="attacker">
  <input name="amount" value="1000">
</form>
<script>
  document.forms[0].submit();
</script>

User's cookies sent automatically!

Defenses

// 1. CSRF Token (csurf)
const csrf = require('csurf');
app.use(csrf());
app.post('/transfer', (req, res) => {
  if (!req.csrfToken()) fail();
});

// 2. SameSite Cookies
response.setHeader(
  'Set-Cookie',
  'session=abc; SameSite=Strict; HttpOnly'
);

// 3. Origin Header Check
if (req.headers.origin !== 'https://bank.com') {
  return res.status(403).send('Forbidden');
}

15 — CORS Misconfiguration

Dangerous Patterns

// ❌ DANGEROUS: Wildcard + Credentials
app.use(cors({
  origin: '*',
  credentials: true
}));

// ❌ DANGEROUS: Blindly Reflect Origin
app.use(cors({
  origin: req.headers.origin,
  credentials: true
}));

// ❌ DANGEROUS: Allow null origin
app.use(cors({
  origin: ['null', 'https://app.com']
}));

Secure Allowlist

const allowedOrigins = [
  'https://app.com',
  'https://app-staging.com'
];

app.use(cors({
  origin: (origin, cb) => {
    if (!origin || allowedOrigins.includes(origin)) {
      cb(null, true);
    } else {
      cb(new Error('Not allowed by CORS'));
    }
  },
  credentials: true,
  methods: ['GET', 'POST'],
  allowedHeaders: [
    'Content-Type',
    'Authorization'
  ]
}));

16 — SSRF (Server-Side Request Forgery)

High-Value SSRF Targets

  • AWS Metadata: 169.254.169.254:80
  • Localhost Services: localhost:8080, localhost:3000
  • Cloud Metadata: Google Cloud, Azure Instance Metadata
  • Internal APIs: Admin panels, internal databases
  • Webhooks: File upload callbacks

Safe Fetching

// VULNERABLE: No validation
const img = await fetch(req.body.url);

// DNS Rebinding Attack:
// First lookup: trusted IP
// Second lookup: 127.0.0.1 (TOCTOU)

// SAFE: Allowlist + DNS validation
const allowedHosts = new Set([
  'images.example.com',
  'cdn.example.com'
]);

const url = new URL(userUrl);
if (!allowedHosts.has(url.hostname)) {
  throw new Error('Not allowed');
}

// Prevent DNS rebinding
const dns = require('dns').promises;
const addr = await dns.resolve4(url.hostname);
if (addr[0].startsWith('127.') ||
    addr[0].startsWith('169.254.')) {
  throw new Error('Private IP not allowed');
}

17 — Broken Access Control (OWASP #1)

Common Failures

  • Missing authorization checks
  • Assuming authentication = authorization
  • Testing with admin account only
  • Insecure direct object references (IDOR)
  • Horizontal privilege escalation
  • User ID in URL/JWT without validation

Secure Implementation

// BAD: Trust JWT decode
const user = jwt_decode(token);
const data = await db.getData(user.id);

// GOOD: Middleware verification
const authMiddleware = (req, res, next) => {
  const verified = verifyJWT(req.token);
  req.user = verified;
  next();
};

// GOOD: Role checking
const requireRole = (roles) => {
  return (req, res, next) => {
    if (!roles.includes(req.user.role)) {
      return res.status(403).send('Forbidden');
    }
    next();
  };
};

app.get('/admin', requireRole(['admin']), handler);

18 — Authentication Failures

What Goes Wrong

  • Weak password requirements
  • No rate limiting on login
  • Passwords visible in logs
  • Plaintext password storage
  • No 2FA for critical accounts
  • Session fixation (no regenerate)
  • Hardcoded credentials

Best Practices

// Password Hashing: bcrypt
const bcrypt = require('bcrypt');
const hash = await bcrypt.hash(password, 12);
const match = await bcrypt.compare(
  password,
  hash
);

// TOTP (Two-Factor)
const speakeasy = require('speakeasy');
const secret = speakeasy.generateSecret();
const verified = speakeasy.totp.verify({
  secret: secret.base32,
  encoding: 'base32',
  token: userToken
});

// Check if password breached
const pwnedPassword =
  require('hibp').search(password);

19 — JWT Security

Dangerous Patterns

// ❌ alg: none attack
const payload = jwt_decode(token);
// Header: {"alg": "none"}
// Attacker can modify payload!

// ❌ RS256 → HS256 confusion
// Server uses RS256 (asymmetric)
// Attacker sends HS256 (symmetric)
// Uses public key as HMAC secret!

// ❌ Weak secret
jwt.sign(payload, 'secret');
// Should be 256+ bit random

Secure Implementation

// SIGN: Specify algorithm explicitly
const token = jwt.sign(
  { userId: user.id, role: user.role },
  process.env.JWT_SECRET,
  {
    algorithm: 'HS256',
    expiresIn: '15m',
    issuer: 'app.com'
  }
);

// VERIFY: Restrict allowed algorithms
const verified = jwt.verify(
  token,
  process.env.JWT_SECRET,
  {
    algorithms: ['HS256'],
    issuer: 'app.com'
  }
);

// STORE: HttpOnly cookie (not localStorage)
res.cookie('jwt', token, {
  httpOnly: true,
  secure: true,
  sameSite: 'strict',
  maxAge: 15 * 60 * 1000
});

20–22 — OAuth, Sessions & IDOR

OAuth PKCE Checklist

  • Generate code_verifier (43-128 chars)
  • Create code_challenge = SHA256(verifier)
  • Send code_challenge in auth request
  • Exchange code + verifier for token
  • Never hardcode client_secret in SPA

IDOR Prevention

// Check ownership
app.get('/posts/:id', async (req, res) => {
  const post = await Post.findById(req.params.id);
  if (post.userId !== req.user.id) {
    return res.status(403).send('Not yours');
  }
  res.json(post);
});

Secure Sessions

// express-session + secure store
const session = require('express-session');
const RedisStore = require('connect-redis');

app.use(session({
  store: new RedisStore({ client: redis }),
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
  cookie: {
    secure: true,     // HTTPS only
    httpOnly: true,   // No JS access
    sameSite: 'strict',
    maxAge: 24 * 60 * 60 * 1000
  }
}));

// Regenerate on login (prevent fixation)
req.session.regenerate((err) => {
  req.session.userId = user.id;
  res.redirect('/dashboard');
});

// Destroy on logout
req.session.destroy();

28 — Content Security Policy (CSP)

Key Directives

default-src 'self'
script-src 'self' 'nonce-{random}'
style-src 'self' 'nonce-{random}'
img-src 'self' https:
font-src 'self' fonts.googleapis.com
connect-src 'self' api.example.com
frame-ancestors 'none'
form-action 'self'
base-uri 'self'
object-src 'none'
require-sri-for script style
upgrade-insecure-requests

Helmet.js Implementation

// Generate nonce
const crypto = require('crypto');
const nonce = crypto.randomBytes(16)
  .toString('base64');

// CSP with Helmet
helmet.contentSecurityPolicy({
  directives: {
    defaultSrc: ["'self'"],
    scriptSrc: ["'self'", `'nonce-${nonce}'`],
    styleSrc: ["'self'", `'nonce-${nonce}'`],
    imgSrc: ["'self'", 'https:'],
    fontSrc: ["'self'", 'fonts.googleapis.com'],
    connectSrc: ["'self'", 'api.example.com'],
    frameAncestors: ["'none'"],
    formAction: ["'self'"],
    baseUri: ["'self'"],
    objectSrc: ["'none'"],
    requireSriFor: ['script', 'style'],
    upgradeInsecureRequests: []
  }
});

// Inject nonce into template
app.get('/', (req, res) => {
  const nonce = crypto.randomBytes(16)
    .toString('base64');
  res.render('index', { nonce });
});

34 — Subresource Integrity (SRI)

How SRI Works

  1. Compute SHA-384 hash of JS/CSS file
  2. Browser computes hash of downloaded file
  3. Hashes don't match → block execution
  4. Protects against CDN compromise

Generate Hash

openssl dgst -sha384 -binary \
  script.js | openssl base64

HTML Implementation

<script src="https://cdn.example.com/lib.js"
  integrity="sha384-abc123..."
  crossorigin="anonymous">
</script>

<link rel="stylesheet"
  href="https://cdn.example.com/style.css"
  integrity="sha384-def456..."
  crossorigin="anonymous">

/* CSP require SRI */
Content-Security-Policy:
  require-sri-for script style

37 — API Security & Rate Limiting

Attack Types

  • Brute force login
  • Password reset spam
  • API abuse / scraping
  • DDoS amplification
  • Enumeration attacks

Rate Limiting Code

// General rate limiter
const rateLimit = require('express-rate-limit');
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 min
  max: 100,                   // 100 requests
  message: 'Too many requests',
  standardHeaders: true,
  legacyHeaders: false
});

app.use(limiter);

// Strict auth limiter
const authLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: 5,  // 5 attempts per minute
  skipSuccessfulRequests: true
});

app.post('/login', authLimiter, async (req, res) => {
  // Validate input with Zod
  const schema = z.object({
    email: z.string().email(),
    password: z.string().min(8)
  });

  const parsed = schema.safeParse(req.body);
  if (!parsed.success) {
    return res.status(400).json(parsed.error);
  }
  // Handle login...
});

38–42 — Backend Security Concepts

Crypto Failures

Use TLS, bcrypt passwords, OWASP-recommended algorithms.

Misconfiguration

Disable debug mode, remove default credentials, minimal permissions.

Software Integrity

Sign binaries, verify checksums, monitor dependencies.

Insecure Deserialization

// VULNERABLE
const obj = JSON.parse(untrusted);

// Use allowlist
const obj = JSON.parse(untrusted, (k, v) => {
  if (!allowed.includes(k)) return undefined;
  return v;
});

Mass Assignment

// VULNERABLE
Object.assign(user, req.body);

// SAFE: Whitelist
const { name, email } = req.body;
user.name = name;
user.email = email;

DoS Prevention

Logging & STRIDE

What to Log

  • Authentication events (login, logout, 2FA)
  • Authorization failures
  • Security exceptions
  • Admin actions

Never Log

  • Passwords, tokens, API keys
  • Credit card numbers
  • PII without good reason
